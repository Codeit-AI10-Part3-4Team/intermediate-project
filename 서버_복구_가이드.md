# GCP VM 환경 복구 가이드

> **대상 환경** Ubuntu 24.04 LTS · JupyterHub · Ollama · venv + pip  
> **최종 수정** 백업 스크립트 실행 시 `backup_meta.txt` 기준으로 갱신  
> **목적** `deploy/` 폴더에 보존된 스냅샷만으로 팀 VM을 완전히 재현한다.

---

## 백업 방법

아래의 명령을 jupyterlab 터미널 상의 저장소 루트 경로에서 실행한다.
jupyterhub의 설정 파일인 `jupyterhub_config.py` 가 root 경로에 있기 때문에,  
반드시 `sudo` 명령을 포함해서 실행해야 백업이 작동한다.

```bash
sudo bash backup_env.sh
```

---

## 경로 표기 약속

> 이 가이드는 **백업 사본 경로**와 **시스템 실제 경로**를 구분해 표기한다. 둘을 혼동하면 복구가 깨진다.
>
> - `~/deploy/...` — **백업 사본**(복구의 *출처*). 구분 편의를 위한 논리 경로일 뿐, 시스템 경로가 아니다.
> - `/root/...`, `/etc/systemd/system/...`, `/opt/jhub-venv/...` — **시스템 실제 경로**(복구의 *배치 대상*).
>
> 즉 복구의 기본 동작은 `cp ~/deploy/<사본> <시스템 실제 경로>`다. 백업 사본 경로를 그대로 운영 경로로 쓰지 않는다.

---

## 사전 체크리스트

복구를 시작하기 전 아래를 확인한다.

- [ ] 새 GCP VM 생성 완료 (Ubuntu 24.04 LTS, e2-standard-4 이상 권장)
- [ ] `backup_meta.txt` 열람 — OS 버전·Python 버전 일치 여부 확인
- [ ] API 키 원본 보유 확인 — **팀 디스코드 고정(pinned) 메시지**에서 확인 (환경 변수 값은 백업에 포함되지 않음)
- [ ] 팀 GitHub 저장소 접근 권한 확인
- [ ] `deploy/` 스냅샷·`backup_env.sh` 로컬 보유 + gcloud CLI 또는 콘솔 SSH 접근 확인

---

## 복구 순서 개요

```
1. OS 설정 + 백업 스냅샷 업로드(SSH)
2. JupyterHub venv 생성 및 패키지 복원
3. JupyterHub 설정 및 서비스 등록
4. Ollama 설치 및 서비스 등록
5. 환경 변수 주입
6. 서비스 기동 및 검증
```

> **주의** 순서를 반드시 지킬 것. 특히 systemd 등록(3~4단계)은  
> venv Python 경로가 확정된 후 진행해야 `ExecStart` 경로 오류를 피할 수 있다.

## Step 1 — OS 설정

[중간 프로젝트 가이드 문서](https://codeit.notion.site/GCP-VM-2376fd228e8d80e0b971cbf0093dad49)를 수행한다.

> OS·시스템 패키지는 학원이 제공한 VM 이미지로 일괄 구성되므로, 별도 apt 복원 단계는 두지 않는다. 추가 OS 설정이 필요하면 위 학원 가이드를 따른다.

## 백업 스냅샷 업로드 및 확인 (SSH)

> **수행 시점** Step 1(OS 설정) 직후, Step 2 이전. 이후 모든 단계가 `~/deploy/` 아래 파일을 참조하므로, 먼저 백업 스냅샷을 새 VM으로 올려야 한다.
> **전제** VM에는 공유 git 클론이 없다. `deploy/`(repo에 커밋됨)와 `backup_env.sh`는 **복구 담당자 본인의 로컬 repo 클론에서** 업로드한다. 프로젝트 소스도 각 팀원이 개별로 클론해 개인 venv에 editable 설치한다(최종 체크리스트 참조).

### 업로드할 파일과 용도

| 업로드 대상(로컬) | 배치 위치(VM) | 용도 |
|---|---|---|
| `deploy/backup_meta.txt` | `~/deploy/` | OS·Python·JupyterHub 버전 메타 — 복구 전 대조 기준 |
| `deploy/venv_freeze/jhub-venv.txt` | `~/deploy/venv_freeze/` | jhub-venv 서드파티 패키지 핀 고정 — **Step 2-2**에서 복원 |
| `deploy/requirements.lock` | `~/deploy/` | 개인 venv 공유 의존성 잠금(editable 제외) — **Step 2-3**에서 복원 |
| `deploy/jupyterhub/jupyterhub_config.py` | `~/deploy/jupyterhub/` | JupyterHub 설정(키 마스킹됨) — **Step 3-2**에서 `/root/`로 배치 |
| `deploy/systemd/jupyterhub.service` | `~/deploy/systemd/` | JupyterHub systemd 유닛 — **Step 3-3**에서 `/etc/systemd/system/`로 배치 |
| `deploy/systemd/ollama.service` | `~/deploy/systemd/` | Ollama systemd 유닛(커스텀 시) — **Step 4**에서 배치 |
| `deploy/apt_packages.txt` | `~/deploy/` | 시스템 패키지 목록(참고용 — 학원 이미지 기반이라 보통 복원 불요) |
| `backup_env.sh` | `~/` | **백업(스냅샷 생성)용** 스크립트 — 복구가 아니라, 복구 완료 후 백업 주기를 다시 잇기 위해 실행 |

### 업로드 방법 (둘 중 하나)

**(A) gcloud CLI — 로컬에서 실행 (권장)**

```bash
# 로컬 작업 디렉터리(=repo 루트) 기준. <VM_NAME>·<ZONE>은 실제 값으로 교체
gcloud compute scp --recurse ./deploy <VM_NAME>:~/deploy --zone <ZONE>
gcloud compute scp ./backup_env.sh   <VM_NAME>:~/       --zone <ZONE>
```

**(B) GCP 콘솔 브라우저 SSH 업로드**

```bash
# 1) 로컬에서 폴더를 하나로 압축
tar czf deploy.tar.gz deploy backup_env.sh

# 2) 콘솔 SSH 창 우상단 톱니바퀴 → 파일 업로드로 deploy.tar.gz 전송

# 3) VM에서 압축 해제 (홈 기준)
tar xzf ~/deploy.tar.gz -C ~/
```

### 업로드 확인 (찾기)

```bash
# 스냅샷이 홈에 자리 잡았는지 구조 확인
ls -R ~/deploy

# 핵심 파일 존재를 일괄 확인 — MISSING이 하나라도 있으면 다음 단계로 진행 금지
for f in venv_freeze/jhub-venv.txt requirements.lock \
         jupyterhub/jupyterhub_config.py systemd/jupyterhub.service; do
    [[ -f ~/deploy/$f ]] && echo "OK       $f" || echo "MISSING  $f"
done
```

> **주의** `~/deploy/...`는 **백업 사본 경로**이며 시스템 실제 경로가 아니다(위 "경로 표기 약속" 참조). 복구의 기본 동작은 이 사본을 시스템 실제 경로로 `cp` 하는 것이다.

### `backup_env.sh` 실행 시점 (혼동 주의)

`backup_env.sh`는 **복구 스크립트가 아니라 백업 스크립트**다. 새 VM에서 *복구 도중* 돌리는 것이 아니라, **복구가 모두 끝나 환경이 정상화된 뒤** 이 VM의 상태를 다시 스냅샷으로 떠 백업 주기를 잇기 위해 실행한다.

```bash
chmod +x ~/backup_env.sh
# /root/jupyterhub_config.py 접근을 위해 sudo 필요 (위 "백업 방법" 참조)
sudo bash ~/backup_env.sh
```

## Step 2 — JupyterHub venv 생성 및 패키지 복원

### 2-1. JupyterHub 전용 venv 생성

`backup_meta.txt`의 `jupyterhub_venv_python` 항목과 Python 버전을 맞춘다.

```bash
# venv 위치는 backup_meta.txt의 jupyterhub_venv 값과 동일하게 생성
sudo mkdir -p /opt/jhub-venv
sudo python3 -m venv /opt/jhub-venv

# 버전 확인 (backup_meta.txt와 대조)
/opt/jhub-venv/bin/python --version
```

### 2-2. pip freeze로 패키지 복원

```bash
# 백업된 freeze 파일 확인
ls ~/deploy/venv_freeze/

# JupyterHub venv 복원
sudo /opt/jhub-venv/bin/pip install --upgrade pip
sudo /opt/jhub-venv/bin/pip install -r ~/deploy/venv_freeze/jhub-venv.txt
```

> **팁** freeze 파일에는 버전이 `==`으로 고정되어 있으므로,  
> Python 마이너 버전(3.10 / 3.11 등)이 다르면 일부 패키지가 설치 실패할 수 있다.  
> 그 경우 오류 패키지만 수동으로 버전 조정 후 재설치한다.

### 2-3. requirements.lock으로 프로젝트 패키지 검증

`requirements.lock`은 editable 패키지가 제외된 공유 의존성 목록이다.  
각 팀원이 개인 venv를 생성한 후 아래를 실행한다.

```bash
# 팀원 개인 실행 — 프로젝트 루트 기준
python3 -m venv .venv
source .venv/bin/activate

pip install --upgrade pip

# 백업된 잠금 파일 확인 (backup_env.sh가 항상 생성)
ls -l ~/deploy/requirements.lock

# 공유 의존성을 핀 고정으로 복원
pip install -r ~/deploy/requirements.lock
# editable 로컬 패키지만 추가 — 잠긴 의존성을 건드리지 않도록 --no-deps
pip install -e . --no-deps
```

## Step 3 — JupyterHub 설정 및 서비스 등록

### 3-1. 설치 확인

```bash
# Step 2에서 venv에 설치했으므로 버전만 확인
/opt/jhub-venv/bin/jupyterhub --version
# backup_meta.txt의 jupyterhub_version 값과 일치하는지 대조
```

### 3-2. 설정 파일 배포

```bash
# 백업 파일을 실제 운영 경로로 복사
# jupyterhub.service의 -f 옵션이 /root/jupyterhub_config.py를 참조한다
sudo cp ~/deploy/jupyterhub/jupyterhub_config.py /root/jupyterhub_config.py

# 마스킹된 API 키 값을 실제 값으로 채운다 (Step 5에서 진행)
sudo nano /root/jupyterhub_config.py
```

**확인 포인트**: 설정 파일 내 `키를 입력하세요` 문구로 마스킹된 항목 목록

```bash
grep -n "키를 입력하세요" /root/jupyterhub_config.py
```

### 3-3. systemd 서비스 등록

```bash
sudo cp ~/deploy/systemd/jupyterhub.service /etc/systemd/system/

sudo systemctl daemon-reload
sudo systemctl enable jupyterhub

# 아직 시작 금지 — 환경 변수 주입(Step 5) 후 기동
```

**확인 포인트**: 서비스 파일의 `ExecStart` 경로가 venv와 일치하는지 반드시 검증한다.

```bash
grep ExecStart ~/deploy/systemd/jupyterhub.service
# 예: ExecStart=/opt/jhub-venv/bin/jupyterhub -f /root/jupyterhub_config.py

# 실제 venv 경로 확인
ls /opt/jhub-venv/bin/jupyterhub
```

경로가 다르면 서비스 파일을 수정한다.

```bash
sudo nano /etc/systemd/system/jupyterhub.service
sudo systemctl daemon-reload
```

## Step 4 — Ollama 설치 및 서비스 등록

```bash
# 공식 설치 스크립트
curl -fsSL https://ollama.com/install.sh | sh

# 서비스 파일 복원 (커스텀 설정이 있는 경우)
if [[ -f ~/deploy/systemd/ollama.service ]]; then
    sudo cp ~/deploy/systemd/ollama.service /etc/systemd/system/
    sudo systemctl daemon-reload
fi

sudo systemctl enable ollama
sudo systemctl start ollama

# 모델 재다운로드 (모델 바이너리는 백업 대상 아님)
# backup_meta.txt 또는 팀 위키에서 사용 모델 목록 확인 후 실행
# 예: ollama pull llama3.1:8b
```

> **주의** Ollama는 localhost-only 바인딩으로 운영 중이므로  
> 방화벽 추가 규칙 없이 JupyterHub와 동일 VM에서 연동 가능하다.

## Step 5 — 환경 변수 주입 ⚠️ 보안 주의

이 환경은 `EnvironmentFile`을 사용하지 않는다. API 키는 `/root/jupyterhub_config.py`의
`c.Spawner.environment` 딕셔너리에 직접 기록되어 각 사용자 서버로 전달된다.

```bash
# 마스킹 문구 위치 확인
grep -n "키를 입력하세요" /root/jupyterhub_config.py

# 실제 값 입력 (키 값은 팀 디스코드 고정 메시지에서 복사)
sudo nano /root/jupyterhub_config.py
```

편집 대상 구문:

```python
c.Spawner.environment = {
    "OPENAI_API_KEY": "키를 입력하세요",   # ← 실제 키 값으로 교체
}
```

```bash
# 파일 권한 강화 (root만 읽기)
sudo chmod 600 /root/jupyterhub_config.py
sudo chown root:root /root/jupyterhub_config.py
```

> **주의** 키 입력 후 파일을 git에 커밋하지 않도록 주의한다.  
> `/root/jupyterhub_config.py`는 `.gitignore` 대상이거나 백업 스크립트가 마스킹 후 추출하므로  
> 원본 파일을 직접 커밋하지 않는다.
>
> **키 출처** 실제 키 값은 팀 디스코드 고정 메시지에서 받는다. 디스코드는 시크릿 스토어가 아니므로(히스토리·검색에 잔존, 항목별 접근제어 없음) 이 키는 **프로젝트 전용·폐기 가능**으로 취급하고, 과제 종료 후 회수(rotate)한다.

## Step 6 — 서비스 기동 및 검증

```bash
# JupyterHub 기동
sudo systemctl start jupyterhub
sudo systemctl status jupyterhub   # Active: running 확인

# 로그 실시간 확인
journalctl -u jupyterhub -f

# Ollama 연결 확인
curl http://localhost:11434/api/tags

# 포트 리스닝 확인 (JupyterHub 기본 8000)
ss -tlnp | grep 8000
```

> **주의(외부 접속)** GCP 기본 네트워크는 8000 포트 인그레스를 열어두지 않는다.  
> 새 VM에서 `http://<VM_외부_IP>:8000`이 안 되면 복구 실패가 아니라 **방화벽 규칙 부재**일 수 있다.  
> 먼저 로컬에서 검증하고(`curl -I http://localhost:8000`), 외부 노출이 필요하면 8000 인그레스 규칙을 별도로 추가한다(노출 범위는 팀/평가 IP로 최소화 권장).

### 최종 체크리스트

- [ ] (로컬) `curl -I http://localhost:8000` → 200/302 응답
- [ ] (외부, 방화벽 규칙 추가 시) `http://<VM_외부_IP>:8000` 접속 → JupyterHub 로그인 화면 표시
- [ ] 각 팀원 계정 로그인 성공
- [ ] Ollama 모델 응답 정상 (`ollama run <모델명>`)
- [ ] 팀 프로젝트 코드 `git clone` 후 editable 패키지 재설치 확인

## 자주 발생하는 문제

| 증상 | 원인 | 해결 |
|------|------|------|
| JupyterHub 시작 실패 | config 파일 경로 불일치 | `journalctl -u jupyterhub` 확인 → `-f` 경로 수정 후 `daemon-reload` |
| `No such file`: jupyterhub | ExecStart venv 경로 불일치 | 서비스 파일의 `ExecStart` 경로 수정 후 `daemon-reload` |
| OPENAI_API_KEY 미전달 | config에 키 값 미입력 | `/root/jupyterhub_config.py`의 `c.Spawner.environment` 재확인 |
| pip install 실패 | Python 버전 불일치 | `backup_meta.txt`의 `jupyterhub_venv_python` 대조 후 venv 재생성 |
| Ollama 연결 안 됨 | 서비스 미기동 | `sudo systemctl start ollama` |
| 모델 없음 | 바이너리 미백업 | `ollama pull <모델명>` 재실행 |

## 참고: 백업에 포함되지 않는 항목

다음은 의도적으로 백업에서 제외되며, 복구 시 별도 처리가 필요하다.

| 항목 | 이유 | 복구 방법 |
|------|------|-----------|
| API 키 값 | 보안 | 팀 디스코드 고정 메시지에서 복사 입력 |
| Ollama 모델 바이너리 | 용량 (수 GB) | `ollama pull` 재실행 |
| venv 디렉토리 자체 | 재생성 가능 | `python3 -m venv` + `pip install -r` |
| 각 팀원 editable 패키지 | 프로젝트 클론 후 재설치 | `pip install -e .` |
| 사용자 데이터/노트북 | 개인 책임 (GitHub 커밋 권장) | 개인 백업 |
