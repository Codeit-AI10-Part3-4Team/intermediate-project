# GCP VM 환경 복구 가이드

> **대상 환경** Ubuntu 22.04 LTS · JupyterHub · Ollama · venv + pip  
> **최종 수정** 백업 스크립트 실행 시 `backup_meta.txt` 기준으로 갱신  
> **목적** `deploy/` 폴더에 보존된 스냅샷만으로 팀 VM을 완전히 재현한다.

---

## 사전 체크리스트

복구를 시작하기 전 아래를 확인한다.

- [ ] 새 GCP VM 생성 완료 (Ubuntu 22.04 LTS, e2-standard-4 이상 권장)
- [ ] `backup_meta.txt` 열람 — OS 버전·Python 버전 일치 여부 확인
- [ ] API 키 원본 보유 확인 (환경 변수 값은 백업에 포함되지 않음)
- [ ] 팀 GitHub 저장소 접근 권한 확인

---

## 복구 순서 개요

```
1. OS 기반 설정
2. 시스템 패키지 설치
3. JupyterHub venv 생성 및 패키지 복원
4. JupyterHub 설정 및 서비스 등록
5. Ollama 설치 및 서비스 등록
6. 환경 변수 주입
7. 서비스 기동 및 검증
```

> **주의** 순서를 반드시 지킬 것. 특히 systemd 등록(4~5단계)은  
> venv Python 경로가 확정된 후 진행해야 `ExecStart` 경로 오류를 피할 수 있다.

## Step 1, 2 — OS 기반 설정 및 APT 패키지 복원

[중간 프로젝트 가이드 문서](https://codeit.notion.site/GCP-VM-2376fd228e8d80e0b971cbf0093dad49)를 따른 후 적용하세요


```bash
# 백업에서 패키지 목록 확인
cat ~/deploy/apt_packages.txt

# 일괄 설치
# (각 패키지 버전은 현재 apt 저장소 최신으로 설치됨 — 버전 고정 필요 시 수동 지정)
xargs sudo apt-get install -y < ~/deploy/apt_packages.txt
```

> **팁** `apt_packages.txt`에 GPU 드라이버 관련 패키지(`nvidia-driver-*`)가  
> 포함된 경우 별도로 GCP 콘솔에서 GPU 유형을 먼저 확인한 뒤 설치한다.

## Step 3 — JupyterHub venv 생성 및 패키지 복원

### 3-1. JupyterHub 전용 venv 생성

`backup_meta.txt`의 `jupyterhub_venv_python` 항목과 Python 버전을 맞춘다.

```bash
# venv 위치는 backup_meta.txt의 jupyterhub_venv 값과 동일하게 생성
sudo mkdir -p /opt/jupyterhub
sudo python3 -m venv /opt/jupyterhub/venv

# 버전 확인 (backup_meta.txt와 대조)
/opt/jupyterhub/venv/bin/python --version
```

### 3-2. pip freeze로 패키지 복원

```bash
# 백업된 freeze 파일 확인
ls ~/deploy/venv_freeze/

# JupyterHub venv 복원
sudo /opt/jupyterhub/venv/bin/pip install --upgrade pip
sudo /opt/jupyterhub/venv/bin/pip install -r ~/deploy/venv_freeze/venv.txt
```

> **팁** freeze 파일에는 버전이 `==`으로 고정되어 있으므로,  
> Python 마이너 버전(3.10 / 3.11 등)이 다르면 일부 패키지가 설치 실패할 수 있다.  
> 그 경우 오류 패키지만 수동으로 버전 조정 후 재설치한다.

### 3-3. requirements.lock으로 프로젝트 패키지 검증

`requirements.lock`은 editable 패키지가 제외된 공유 의존성 목록이다.  
각 팀원이 개인 venv를 생성한 후 아래를 실행한다.

```bash
# 팀원 개인 실행 — 프로젝트 루트 기준
python3 -m venv .venv
source .venv/bin/activate

pip install --upgrade pip
pip install -r ~/deploy/requirements.lock

# editable 패키지 재설치 (프로젝트 클론 후)
pip install -e .
```

## Step 4 — JupyterHub 설정 및 서비스 등록

### 4-1. 설치 확인

```bash
# Step 3에서 venv에 설치했으므로 버전만 확인
/opt/jupyterhub/venv/bin/jupyterhub --version
# backup_meta.txt의 jupyterhub_version 값과 일치하는지 대조
```

### 4-2. 설정 파일 배포

```bash
sudo mkdir -p /etc/jupyterhub

# 백업 파일 복사
sudo cp ~/deploy/jupyterhub/jupyterhub_config.py /etc/jupyterhub/

# 마스킹된 API 키 값을 실제 값으로 채운다 (Step 6에서 진행)
sudo nano /etc/jupyterhub/jupyterhub_config.py
```

**확인 포인트**: 설정 파일 내 빈 문자열(`""`)로 마스킹된 항목 목록

```bash
grep -n '= ""' /etc/jupyterhub/jupyterhub_config.py
```

### 4-3. systemd 서비스 등록

```bash
sudo cp ~/deploy/systemd/jupyterhub.service /etc/systemd/system/

sudo systemctl daemon-reload
sudo systemctl enable jupyterhub

# 아직 시작 금지 — 환경 변수 주입(Step 6) 후 기동
```

**확인 포인트**: 서비스 파일의 `ExecStart` 경로가 venv와 일치하는지 반드시 검증한다.

```bash
grep ExecStart ~/deploy/systemd/jupyterhub.service
# 예: ExecStart=/opt/jupyterhub/venv/bin/jupyterhub -f /etc/jupyterhub/jupyterhub_config.py

# 실제 venv 경로 확인
ls /opt/jupyterhub/venv/bin/jupyterhub
```

경로가 다르면 서비스 파일을 수정한다.

```bash
sudo nano /etc/systemd/system/jupyterhub.service
sudo systemctl daemon-reload
```

## Step 5 — Ollama 설치 및 서비스 등록

```bash
# 공식 설치 스크립트
curl -fsSL https://ollama.com/install.sh | sh

# 서비스 파일 복원 (커스텀 설정이 있는 경우)
if [[ -f ~/deploy/systemd/ollama.service ]]; then
    sudo cp ~/deploy/systemd/ollama.service /etc/systemd/system/
    sudo systemctl daemon-reload
fi

sudo systemctl enable ollama
sudo systemctl start ollama

# 모델 재다운로드 (모델 바이너리는 백업 대상 아님)
# backup_meta.txt 또는 팀 위키에서 사용 모델 목록 확인 후 실행
# 예: ollama pull llama3.1:8b
```

> **주의** Ollama는 localhost-only 바인딩으로 운영 중이므로  
> 방화벽 추가 규칙 없이 JupyterHub와 동일 VM에서 연동 가능하다.

## Step 6 — 환경 변수 주입 ⚠️ 보안 주의

백업된 `deploy/systemd/env/jupyterhub.env`에는 **키 이름만** 남아 있다.  
실제 값은 팀 디코에서 검색해 가져온다.

```bash
# 키 이름 확인
cat ~/deploy/systemd/env/jupyterhub.env
# 출력 예:
# JUPYTERHUB_CRYPT_KEY=
# OPENAI_API_KEY=
# ...

# 실제 값 채우기
sudo cp ~/deploy/systemd/env/jupyterhub.env /etc/jupyterhub/jupyterhub.env
sudo nano /etc/jupyterhub/jupyterhub.env
# → 각 키에 실제 값 입력 후 저장

# 파일 권한 강화 (owner read-only)
sudo chmod 600 /etc/jupyterhub/jupyterhub.env
sudo chown root:root /etc/jupyterhub/jupyterhub.env
```

### systemd에서 env 파일 참조 확인

`deploy/systemd/jupyterhub.service` 내에 아래 구문이 있는지 확인:

```ini
[Service]
EnvironmentFile=/etc/jupyterhub/jupyterhub.env
```

없으면 서비스 파일에 추가한다.

## Step 7 — 서비스 기동 및 검증

```bash
# JupyterHub 기동
sudo systemctl start jupyterhub
sudo systemctl status jupyterhub   # Active: running 확인

# 로그 실시간 확인
journalctl -u jupyterhub -f

# Ollama 연결 확인
curl http://localhost:11434/api/tags

# 포트 리스닝 확인 (JupyterHub 기본 8000)
ss -tlnp | grep 8000
```

## 자주 발생하는 문제

| 증상 | 원인 | 해결 |
|------|------|------|
| JupyterHub 시작 실패 | EnvironmentFile 경로 불일치 | `journalctl -u jupyterhub` 확인 후 경로 수정 |
| `No such file`: jupyterhub | ExecStart venv 경로 불일치 | 서비스 파일의 `ExecStart` 경로 수정 후 `daemon-reload` |
| 인증 오류 | API 키 값 누락 | `/etc/jupyterhub/jupyterhub.env` 값 재확인 |
| pip install 실패 | Python 버전 불일치 | `backup_meta.txt`의 `jupyterhub_venv_python` 대조 후 venv 재생성 |
| Ollama 연결 안 됨 | 서비스 미기동 | `sudo systemctl start ollama` |
| 모델 없음 | 바이너리 미백업 | `ollama pull <모델명>` 재실행 |

## 참고: 백업에 포함되지 않는 항목

다음은 의도적으로 백업에서 제외되며, 복구 시 별도 처리가 필요하다.

| 항목 | 이유 | 복구 방법 |
|------|------|-----------|
| API 키 값 | 보안 | 팀 디코에서 찾아 입력 |
| Ollama 모델 바이너리 | 용량 (수 GB) | `ollama pull` 재실행 |
| venv 디렉토리 자체 | 재생성 가능 | `python3 -m venv` + `pip install -r` |
| 각 팀원 editable 패키지 | 프로젝트 클론 후 재설치 | `pip install -e .` |
| 사용자 데이터/노트북 | 개인 책임 (GitHub 커밋 권장) | 개인 백업 |
